# Contributing to Atlas Terminal
Thank you for your interest in contributing to Atlas Terminal! We're excited to have you on board.

## Getting Started
Before you start contributing, please make sure you have the following:

* Node.js 18 or higher installed on your machine
* A free Groq API key from [console.groq.com](https://console.groq.com/keys)
* A Tavily API key from [app.tavily.com](https://app.tavily.com) (required for web search feature)
	+ To obtain a Tavily API key, follow these steps:
		1. Create an account on [app.tavily.com](https://app.tavily.com)
		2. Go to the API keys section and create a new API key
		3. Copy the API key and store it securely
* A code editor or IDE of your choice

## Setting up your development environment
To set up your development environment, follow these steps:

1. Install the required dependencies by running `npm install` in the project root.
2. Set up your Groq API key by running `atlas config` and following the prompts.
3. Set up your Tavily API key by running `atlas config` and following the prompts (you will be asked for your Tavily API key after setting up your Groq API key).
4. Start the development server by running `npm run dev`.

Note: You will need to have Node.js 18 or higher installed on your machine to run the development server.

## Code of Conduct
We follow the standard open-source code of conduct. Please be respectful and considerate of others when interacting with the project.

## How to Contribute
There are several ways to contribute to Atlas Terminal:

### 1. Issues
If you find a bug or have a feature request, please create an issue on our GitHub repository. Make sure to include as much detail as possible, including:

* A clear description of the issue or feature request
* Any relevant code or logs
* Steps to reproduce the issue (if applicable)

### 2. Pull Requests
If you'd like to submit a pull request, please follow these steps:

* Fork the Atlas Terminal repository
* Create a new branch for your changes
* Make your changes and commit them with a clear and descriptive commit message
* Submit a pull request to the main repository

### 3. Code Review
We use code reviews to ensure that all changes meet our standards. If you'd like to review code, please comment on the relevant pull request with your feedback.

## Code Style
We use the following code style guidelines:

* JavaScript: 2 spaces for indentation, semicolons at the end of statements
* Markdown: Use headings, bold and italic text, and links as needed

## Commit Message Guidelines
We use the following commit message guidelines:

* Use the imperative mood (e.g. "Fix bug" instead of "Fixed bug")
* Keep the first line short and descriptive (less than 50 characters)
* Use a blank line to separate the brief summary from the body
* Use bullet points to list changes or features
